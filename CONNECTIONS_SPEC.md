# EasyStock — Retailer ↔ Wholesaler Connection Spec

> This document defines the Firestore collection, status lifecycle, business rules,
> and Flutter implementation plan for the retailer–wholesaler connection feature.
> Read alongside CLAUDE.md.

---

## 1. Overview

A **retailer** (mobile app) sends a connection request to a **wholesaler**.  
The wholesaler (web/mobile) reviews and accepts or rejects it.  
Once **approved**, the retailer sees the wholesaler in their **My Wholesalers** tab
and can browse stock + place orders.

---

## 2. Firestore Collection

### Collection name: `connections`

**Document ID:** auto-generated by Firestore  
**Path:** `/connections/{connectionId}`

### Fields

| Field | Type | Description |
|---|---|---|
| `retailerId` | string | UID of the retailer (from `retailers` collection) |
| `wholesalerId` | string | Document ID of the wholesaler (from `wholesalers` collection) |
| `status` | string | `pending` · `approved` · `rejected` · `blocked` |
| `requestedAt` | timestamp | When the retailer sent the request |
| `respondedAt` | timestamp | When the wholesaler approved/rejected (null until then) |
| `retailerName` | string | Snapshot: retailer's shop name at time of request |
| `retailerCity` | string | Snapshot: retailer's city at time of request |
| `wholesalerName` | string | Snapshot: wholesaler's business name at time of request |
| `rejectionReason` | string | Optional — filled by wholesaler on rejection |
| `blockedAt` | timestamp | Optional — when wholesaler blocked the retailer |
| `blockedBy` | string | `wholesaler` or `retailer` (future use) |

### Example Document

```
/connections/abc123xyz
  retailerId:           "uid_of_retailer"
  wholesalerId:         "dummy_wholesaler_03"
  status:               "approved"
  requestedAt:          June 26 2026 10:00:00 UTC+5
  respondedAt:          June 26 2026 11:30:00 UTC+5
  retailerName:         "Ali Paint Store"
  retailerCity:         "Lahore"
  wholesalerName:       "Demo Wholesaler 3"
  wholesalerCity:       "Lahore"
  wholesalerBrands:     "Brighto · ICI · Berger"
  wholesalerPhone:      "0300-1234567"
  wholesalerAddress:    "Main Market, Gulberg"
  wholesalerEmail:      "demo@wholesaler.com"
  wholesalerColorIndex: 2
  rejectionReason:      ""
```

> **Note:** wholesaler snapshot fields are stored at request time so My Wholesalers,
> Home tile list, and detail screen never need a secondary Firestore fetch.

## 3. Status Lifecycle

```
[Retailer taps Connect]
        │
        ▼
    PENDING  ──── Wholesaler rejects ────► REJECTED
        │                                    │
        │                                    └── Retailer can re-request after 7 days
        │
   Wholesaler approves
        │
        ▼
    APPROVED ──── Wholesaler blocks ────► BLOCKED
        │                                    │
        └── Both sides retain history        └── Retailer sees "blocked" state
                                                  Cannot place orders
```

### Status meanings for retailer app

| Status | Discover tab | My Wholesalers tab | Can browse stock | Can order |
|---|---|---|---|---|
| `none` (no doc) | Shows "Connect" button | Not shown | ✗ | ✗ |
| `pending` | Shows "Requested" pill | Not shown | ✗ | ✗ |
| `approved` | Hidden (moved to My Wholesalers) | ✅ Shown | ✅ | ✅ |
| `rejected` | Shows "Connect" button again | Not shown | ✗ | ✗ |
| `blocked` | Shows "Blocked" pill | Not shown | ✗ | ✗ |

## 4. Firestore Queries Needed

### Retailer app — check connection status for a wholesaler
```
connections
  .where('retailerId', == currentUserId)
  .where('wholesalerId', == wholesalerId)
  .limit(1)
```

### Retailer app — My Wholesalers tab (approved only)
```
connections
  .where('retailerId', == currentUserId)
  → filter client-side: status == 'approved'
```
→ Wholesaler snapshot fields (name, city, brands, phone, address, email, colorIndex)
  are stored directly in the connection doc — no secondary wholesaler fetch needed.

### Retailer app — all connections (single stream, used for everything)
```
connections
  .where('retailerId', == currentUserId)
```
→ `ConnectionsProvider` streams this once and builds:
  - `statusMap`: wholesalerId → status  (used by Discover tab for pills/buttons)
  - `connectionData`: wholesalerId → full doc  (used by Home + Profile approved count/list)
  - `approvedCount`: int  (Home + Profile stat cards)
  - `approvedConnections`: List  (Home My Wholesalers horizontal list)

### Wholesaler app/web — incoming requests
```
connections
  .where('wholesalerId', == wholesalerId)
  .where('status', == 'pending')
  .orderBy('requestedAt', descending: true)
```

## 5. Firestore Indexes Required

Create these **composite indexes** in Firestore Console → Indexes:

| Collection | Field 1 | Field 2 | Field 3 | Query scope |
|---|---|---|---|---|
| `connections` | `retailerId` (ASC) | `status` (ASC) | — | Collection |
| `connections` | `retailerId` (ASC) | `wholesalerId` (ASC) | — | Collection |
| `connections` | `wholesalerId` (ASC) | `status` (ASC) | `requestedAt` (DESC) | Collection |

## 6. Security Rules (Key Points)

```
// Retailer can create a connection (request)
// Only the retailer who owns it can read their own connections
// Wholesaler can read connections where wholesalerId == their uid
// Only wholesaler can update status to 'approved' / 'rejected' / 'blocked'
// Neither side can delete a connection document

match /connections/{connectionId} {
  allow create: if request.auth != null
    && request.resource.data.retailerId == request.auth.uid
    && request.resource.data.status == 'pending';

  allow read: if request.auth.uid == resource.data.retailerId
    || request.auth.uid == resource.data.wholesalerId;

  allow update: if request.auth.uid == resource.data.wholesalerId
    && request.resource.data.retailerId == resource.data.retailerId; // cannot change retailer
}
```

## 8. `connections` Provider Sketch

```dart
class ConnectionsProvider extends ChangeNotifier {
  final _db = FirebaseFirestore.instance;
  
  // Map of wholesalerId → connection status string
  Map<String, String> _statusMap = {};
  // Map of wholesalerId → connectionId (needed to update later)
  Map<String, String> _idMap = {};
  
  Map<String, String> get statusMap => _statusMap;

  StreamSubscription? _sub;

  void listen(String retailerId) {
    _sub?.cancel();
    _sub = _db
        .collection('connections')
        .where('retailerId', isEqualTo: retailerId)
        .snapshots()
        .listen((snap) {
      _statusMap = {};
      _idMap = {};
      for (final doc in snap.docs) {
        final d = doc.data();
        final wId = d['wholesalerId'] as String;
        _statusMap[wId] = d['status'] as String;
        _idMap[wId] = doc.id;
      }
      notifyListeners();
    });
  }

  Future<void> sendRequest({
    required String retailerId,
    required String retailerName,
    required String retailerCity,
    required String wholesalerId,
    required String wholesalerName,
  }) async {
    await _db.collection('connections').add({
      'retailerId':     retailerId,
      'wholesalerId':   wholesalerId,
      'status':         'pending',
      'requestedAt':    FieldValue.serverTimestamp(),
      'respondedAt':    null,
      'retailerName':   retailerName,
      'retailerCity':   retailerCity,
      'wholesalerName': wholesalerName,
      'rejectionReason': '',
    });
  }

  void dispose() {
    _sub?.cancel();
    super.dispose();
  }
}
```

## 10. Key Constraints

- One retailer can have at most **one connection doc per wholesaler** — enforce on write (check for existing doc first).
- A `rejected` connection can be re-requested — create a **new doc**, don't reuse/update the old one.  
- A `blocked` connection can only be unblocked by the **wholesaler**.
- Connection doc holds **snapshot** of names at request time — this protects display if the wholesaler later renames their business.
- Never delete connection docs — they are the audit trail for the relationship history.
